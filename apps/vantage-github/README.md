# vantage-github

A Vantage UI example that turns GitHub Actions pipelines into browsable tables through the
`gh` CLI — and **generates new inventory at runtime**. Pick a workflow, hit a button, and a
dedicated runs table whose columns are that workflow's *steps* appears in the sidebar, live.

## What it does

1. **Cache Efficiency** — lists every build workflow in `romaninsh/vantage`. An *augmentation*
   pass enriches each row with the workflow's steps (discovered from its latest successful
   run).
2. **Generate runs table** (row action) — runs a command that pulls the workflow's latest
   successful run, discovers its steps, and **writes a new table + page** whose columns are
   those steps. Each cell is that step's duration in seconds for the run on that row. The
   command also registers the page in the left menu. The viewer **hot-reloads** the new
   files — no restart — so the table shows up under "Workflow Runs" immediately.
3. **Dashboard** — regenerated by the same command: one **stacked-area chart per workflow**,
   distributing runs along X (one point per run) and stacking each step as a colored layer,
   so the top edge is total runtime. No axis labels (dense tiles). Auto-assigned colors.

This is a deliberately **self-mutating inventory**: the generated `table/runs-*.yaml`,
`page/runs-*.yaml` and the menu section are real files you commit. Configure once, commit,
done.

## How it works

Vantage can't create tables or menu items from Rhai (Rhai only creates *rows* in existing
tables). The only way to a brand-new, per-workflow table is to **write inventory files** — and
the viewer's file watcher picks up newly created `table/`, `page/` and edited `menu/` files
live (`EventKind::Create` → `TopologyChanged` → sidebar re-renders). A `kind: terminal` action
runs a script that does exactly that; its working directory defaults to the inventory root, so
it knows where to write.

```
datasource/
  gh-stats.yaml            # type: cmd → scripts/gh-rust-caching-stats

table/
  gh-workflows.yaml        # workflows + steps augmentation (detail pass)
  runs-<slug>.yaml         # GENERATED per workflow — columns = that workflow's steps

page/
  cache-efficiency.yaml    # workflows table + "Generate runs table" row action
  runs-<slug>.yaml         # GENERATED per workflow — tableview over runs-<slug>
  dashboard.yaml           # GENERATED — one stacked-area chart per runs table

action/
  scaffold-runs.yaml       # kind: terminal → runs `scaffold-runs`, writes the files above

menu/
  left.yaml                # static "Cache Efficiency" + GENERATED "Dashboard" + "Workflow Runs"
```

The dashboard uses `chart_type: stacked_area` — a chart kind added to vantage-ui for this
example. It takes a `series:` list (one column per layer, stacked to a cumulative total) and
`labels: false` to drop axis text for dense tiles; colors are auto-assigned across the layers.

## The command (`scripts/gh-rust-caching-stats`)

A plain Python wrapper around `gh`. Subcommands:

| Subcommand                              | Returns                                                            |
| --------------------------------------- | ----------------------------------------------------------------- |
| `workflows [repo]`                      | JSON list of build workflows                                      |
| `runs <wf> [repo] [--success]`          | JSON list of runs (`--success` filters to successful)             |
| `workflow-steps <wf> [repo]`            | canonical step list (names + slugs) from the latest successful run |
| `run-steps <run_id> [repo]`             | flat map `{ step_slug: duration_seconds }` for one run            |
| `scaffold-runs <wf> [repo] --name <label> --inventory <dir>` | writes `table/runs-<slug>.yaml`, `page/runs-<slug>.yaml`, updates `menu/left.yaml` |

Step data comes from `gh run view <run_id> --json jobs` (`jobs[].steps[]`); a step's duration
is `completedAt - startedAt`. Columns are keyed by a **slug of the step name** (not its number)
so a column lines up across every run even when step numbers shift.

## Tables

### gh-workflows

`workflows` → build pipelines, augmented per-row by `workflow-steps`:

| Column      | Type   | Notes                                            |
| ----------- | ------ | ------------------------------------------------ |
| id          | int    | [id]                                             |
| name        | string | [title, searchable]                              |
| state       | string | active/disabled                                  |
| step_count  | int    | augmentation — # of steps in the latest run      |
| steps       | string | augmentation — comma-joined step names           |

### runs-&lt;slug&gt; (generated)

One per workflow you scaffold. Lists the workflow's **successful** runs; columns after
`created_at` are the steps:

| Column        | Type     | Notes                                         |
| ------------- | -------- | --------------------------------------------- |
| run_id        | int      | [id]                                          |
| run_number    | int      |                                               |
| head_branch   | string   | [title]                                       |
| conclusion    | string   |                                               |
| created_at    | datetime |                                               |
| &lt;step slug&gt;   | int      | one per step — duration in seconds (detail pass) |

## Notes

- Hardcoded to the `romaninsh/vantage` repo (the workflow id is baked into each generated
  table).
- The generated runs tables list **successful runs only** — they're the runs guaranteed to
  have executed every step (the column set). Step columns use the *latest* successful run's
  steps; earlier runs are assumed to share that step setup.
- Re-running "Generate runs table" on the same workflow is **idempotent**: the table/page are
  overwritten and the menu section is regenerated (never duplicated). Files are written
  atomically (temp + rename) so the watcher never sees a half-written YAML.
- Each run costs one `gh run view --json jobs` call in the detail pass, hydrated lazily as
  rows scroll into view.
